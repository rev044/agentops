# Learning: v2.37.0 Release Process — Large Batches Hide Good Work

## What We Learned

### L1: A green gate stack does not make a 227-commit release easy to reason about
This cut passed the full local release gate and the latest remote `Validate`
workflow on `main`, but the range still spanned 227 non-merge commits over
three days. The work was real and shippable, yet the release story had to be
compressed aggressively because too many unrelated improvements landed between
tags.

### L2: Windows only became a credible target once install, smoke, and runtime behavior shipped together
The Windows work was not just `install-ao.ps1`. The release also needed native
PowerShell smoke coverage and a real runtime bug fix for overnight process
liveness before "Windows support" became a defensible release claim.

### L3: Karpathy-style index-first navigation is a low-risk import path
AgentOps already had the raw ingredients for a repo-local knowledge wiki. The
big lift was adding the discovery surface and ergonomic glue: `INDEX.md`,
`LOG.md`, wiki directories, and ranking boosts that treat index-linked pages as
first-class context.

## Why It Matters

This release contains strong product work, but large ranges make it harder to
audit, easier to under-document, and slower to bisect when regressions appear.
The main release lesson is process: keep shipping windows smaller so the public
story and the debugging story stay aligned.

## Recommendations

1. Keep future minor releases closer to the repo guidance of 15 commits unless
   there is a clear reason to batch.
2. Treat Windows support claims as a three-part contract: install path, smoke
   coverage, and runtime behavior.
3. Prefer the index-first wiki pattern when importing ideas from adjacent tools;
   it upgrades ergonomics without replacing AgentOps' existing flywheel model.
