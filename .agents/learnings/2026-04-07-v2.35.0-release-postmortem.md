# Learning: v2.35.0 Release Process — CI Failure Cascade

## What We Learned

### L1: Local CI gate and remote CI have different failure surfaces
The local `ci-local-release.sh` gate caught failures (secret scan, skill schema, JSON flag permissions) that remote CI didn't test, and remote CI caught failures (doc-release-gate, skill-lint line limit, cli-docs-parity) that the local gate didn't flag. **Neither gate alone is sufficient.** The release required 4 tag rewrites because failures surfaced incrementally.

### L2: Environment variable leaks are invisible until shuffle exposes them
`AGENTOPS_RPI_RUNTIME=bushido` in the host environment caused 17+ test failures that only appeared when running tests locally. Tests passed in CI (clean env) but failed locally. Fix: `t.Setenv` in each affected test. **Prevention: add env var isolation to test scaffolding helpers, not individual tests.**

### L3: Codex hooks.json uses a flat array format, not event-keyed
Claude Code: `{"hooks": {"SessionStart": [...]}}`. Codex: `{"hooks": [{"event": "SessionStart", ...}]}`. The install script must render both formats from the same hook definitions.

### L4: docs/CHANGELOG.md is a separate file that must be synced
Root `CHANGELOG.md` and `docs/CHANGELOG.md` are independent copies. Updating one doesn't update the other. The doc-release-gate checks `docs/CHANGELOG.md` specifically.

### L5: Skill line limits (250 lines) are hard CI gates
Adding 5 lines to using-agentops SKILL.md (from 250 to 255) broke skill-lint. Must check `wc -l` before committing skill changes.

### L6: Coverage percentage CI gates blocked architectural refactors
coverage-ratchet and cmd/ao coverage floor checks failed repeatedly during internal/ extraction waves without catching any real bugs. Removed entirely — tests catch regressions, not coverage percentages.

### L7: Retagging releases is expensive
v2.35.0 was tagged and force-pushed 3 times as post-tag CI failures were fixed. **Prevention: run the full CI pipeline (local + remote) before creating the first tag.** The release skill should enforce a "CI must be green on HEAD before tagging" gate.

## Why It Matters

This release took ~3 hours of CI fix iterations that should have been caught by a single comprehensive pre-release validation. The pattern: fix one failure, push, discover the next failure, repeat. Each cycle costs ~5 minutes of CI wait time plus context overhead.

## Recommendations

1. **Add a pre-tag CI checklist** to the release skill that runs both local gate AND checks remote CI status before allowing `git tag`
2. **Add `docs/CHANGELOG.md` sync** to the release skill workflow (Step 9)
3. **Add `wc -l` check** to skill editing hooks (PreToolUse on Edit for SKILL.md files)
4. **Add `scripts/generate-cli-reference.sh`** to the release skill pre-flight when Go files changed
5. **Add env var isolation** to `TestMain` or a package-level test helper instead of per-test `t.Setenv`
