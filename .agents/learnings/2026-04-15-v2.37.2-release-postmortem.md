# Learning: Hotfix Releases Need Explicit Scope and Evidence When the Internal Range Is Large

## What We Learned

### L1: A patch release can still be the right external contract when the work is mostly hardening
The `2.37.2` range was broad, but the user-facing contract stayed centered on
operational hardening: stronger validation, safer hooks, and reliability fixes
instead of a new top-level product story.

### L2: Release prep has to track every shipped version surface
The release touched not only changelogs and plugin manifests, but also
comparison docs, curated notes, and the internal audit record. Missing any of
those leaves the repo telling conflicting version stories.

### L3: Large hotfix ranges need an audit trail, not just a tag
Once a patch release spans dozens of commits, the audit file becomes the place
to explain why the semver choice is still deliberate and what evidence was used
to ship it.

## Why It Matters

Release trust comes from matching the external promise, the shipped version
surfaces, and the validation evidence. When those stay aligned, even a broad
operational-hardening patch can remain understandable and defensible.

## Recommendations

1. Decide the public semver based on user-facing contract change, not raw
   commit count alone.
2. Keep the release notes concise, but use the audit file to preserve scope,
   warnings, and evidence.
3. Re-run the full release gate against the final target version before tagging
   so SBOM and security artifacts match the shipped version.
