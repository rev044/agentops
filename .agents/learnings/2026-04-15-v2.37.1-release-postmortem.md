# Learning: v2.37.1 Dream Work Needed Better Output, Not Just More Runtime

## What We Learned

### L1: Dream only became release-worthy once it produced executable morning work
The biggest jump in Dream usefulness did not come from making it run longer. It
came from making the overnight run leave behind ranked morning packets with
evidence, target files, and exact follow-up commands.

### L2: Extra overnight runtime only pays off when it is spent on corroboration
The long-haul path became defensible once it turned into a bounded
evidence-corroboration pass for weak output. Blindly stretching the runtime
would have added cost without improving the morning handoff.

### L3: Headless council adapters need real CLI-contract proof
The Claude council bug existed because the runner assumed the wrong JSON
contract. The fix required testing against Claude's actual headless envelope,
not just validating our own expected schema.

## Why It Matters

Dream is now closer to the intended product shape: a short default overnight
pass that produces useful morning work, with optional extra runtime only when
the first answer is weak. That is a stronger product story and a better
operator story than "Dream runs for longer."

## Recommendations

1. Measure Dream by morning-packet quality, not by wall-clock duration.
2. Keep the default overnight path short and reserve long-haul time for weak
   or ambiguous output.
3. Add real CLI-contract coverage for every headless council runner, not just
   Codex.
