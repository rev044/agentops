Operational layer for coding agents

Coding agents do not do their own bookkeeping.

AgentOps gives Claude Code, Codex, Cursor, and OpenCode repo-local memory, validation gates, and repeatable flows so every session starts where the last one left off.

Install AgentOps How it works View source

The loop that compounds.

Agent sessions are disposable. The repo should not be. AgentOps turns raw session signal into reusable context, stronger gates, and better next work.

Bookkeeping that survives chat history

Findings, learnings, handoffs, and decisions land in repo-local files your next agent can search, rank, and reuse.

Validation before ship

/pre-mortem, /vibe, and /council challenge plans and code before regressions become commits.

Flows instead of improvisation

/research, /implement, /rpi, and /evolve route work through discovery, implementation, validation, and learning.

One primitive or the full cycle.

Start with the smallest useful motion. Keep the same operating model when the work grows.

Review a PR

Independent judges surface disagreement and converge on a verdict before you merge.

> /council validate this PR

Ship a scoped change

Discovery, planning, execution, validation, and post-mortem run as one delivery lane.

> /rpi "add retry backoff"

Improve the repo

Measure goals, pick the worst gap, run the lifecycle, and keep the ratchet moving.

> /evolve

Built for real agent work.

AgentOps uses the same surfaces it ships: issue-backed work, validation gates, RPI phases, and post-work learning capture.

Parallel work stays legible

Fresh workers, tracked epics, file deltas, and status lines keep multi-agent execution reviewable.

Completion has evidence

Crank reports waves, regressions, tests, coverage, commits, and next validation steps before the loop closes.
